## 一、问题本质：为什么当前方法撞上了天花板

### 1.1 当前方法的根本缺陷

当前系统把复盘建模为 **45 个独立的分类问题**：

```
Image(segment_i) → Classifier → Label_i
```

每一步之间零信息交换。这意味着：

- **No error correction**: 分类器在第 10 步犯的错，对第 11 步没有任何警告
- **No context**: 第 10 步不知道第 9 步是什么，无法利用"R 后面不太可能接 R'"这样的先验
- **No consistency**: 45 个预测拼起来可能是一条不合法的走法序列（比如 Cross 做到一半去做了 PLL）

### 1.2 L/R 混淆只是表象

L/R 混淆确实是当前最大的错误来源，但即使完美解决了 L/R 区分，准确率也只能从 60% → 72% 左右（因为 U/R 混淆、方向判断等其他问题依然存在）。

**真正的瓶颈是：没有利用魔方的内在约束。**

### 1.3 信息量分析

一个 45 步的 CFOP 解法，理论信息量：

```
候选动作 = U, U', U2, D, D', D2, R, R', R2, L, L', L2, F, F', F2, f, f', r, r'
                                                           ≈ 19 种（不含 B）
信息量 = 45 × log2(19) ≈ 45 × 4.25 ≈ 191 bits
```

而一帧 ROI 图像（960×540 缩放到 480×270）提供的有效信息远不止 191 bits。问题不是信息不够，而是 **提取和利用方式太原始**。

## 二、新范式：有状态的序列搜索

### 2.1 范式对比

| | 旧: 独立分类 | 新: 全局搜索 |
|--|------------|------------|
| **单位** | 每帧独立决策 | 全序列联合优化 |
| **状态** | 无 (Stateless) | 有 (Stateful，维护虚拟魔方) |
| **错误处理** | 错了就错了 | 前后步一致性检查可纠错 |
| **信息利用** | 仅视觉特征 | 视觉 + 魔方规则 + CFOP 结构 |
| **适合的准确率** | > 90% 才可用 | **60% 就能 work** |

## 四、各模块详细设计

### 4.1 魔方状态机 (`cube_state.py`)

**职责**: 维护 54 个贴纸的颜色，支持转动操作和状态查询。

**表示方法**: 54 元素数组，按 U(0-8), R(9-17), F(18-26), D(27-35), L(36-44), B(45-53) 排列。

```
面的编号 (以 U 面为例):
  0 1 2
  3 4 5     ← U[0..8]
  6 7 8
```

**必须支持的操作**:

| 操作 | 说明 | 备注 |
|------|------|------|
| `U, U', U2` | 上层 | 标准 |
| `D, D', D2` | 下层 | 标准 |
| `R, R', R2` | 右层 | 标准 |
| `L, L', L2` | 左层 | 标准 |
| `F, F', F2` | 前层 | 标准 |
| `B, B', B2` | 后层 | 虽然解法不含 B，但状态机需要完整 |
| `x, y, z` | 整体旋转 | splits 中出现 x, y |
| `r, f` | 宽转 | r = R + M', f = F + S |

**关键方法**:
- `apply_move(move_str)`: 解析并执行一个动作字符串
- `get_visible_colors()`: 返回相机能看到的贴纸颜色（B面全部 + L面部分 + U面边缘）
- `clone()`: 深拷贝（搜索分支时用）
- `is_solved()`: 判断是否还原

**实现策略**: 不造轮子。先调研 Python 现有的魔方库（如 `rubik-cube`、`pycuber`），如果能直接用就不自己写。

### 4.2 视觉概率模型（改造 `template_cnn.py`）

**当前**: 输出硬标签 `R`。
**改造后**: 输出概率分布 `{R: 0.35, L: 0.28, U: 0.22, F: 0.10, D: 0.05}`。

**改造最小化**:
- KNN 本身可以输出 K 个邻居的标签分布，不需要训练新模型
- 如果 K=5 且 3 个邻居是 R、1 个是 L、1 个是 U → 输出 `{R: 0.6, L: 0.2, U: 0.2}`

**方向区分**: 当前模型不区分 `R` 和 `R'`。这是一个后续问题。初版搜索不区分方向，先把"面"搜对。

> [!NOTE]
> **简化假设 (初版)**: 先不区分方向和双转。候选空间从 19 种缩减为 5 种 (U/D/R/L/F)。
> 方向由搜索引擎通过一致性检查来推断（检查颜色是顺时针还是逆时针移动了）。

### 4.3 视觉一致性评分器

这是整个系统的 **核心创新**，也是最需要精心设计的模块。

**核心问题**: 给定一个虚拟魔方状态 S 和一帧视频 F，如何评估它们的"吻合度"？

#### 方案 A: 颜色直方图比较（最简单，推荐初版用）

```
1. 从虚拟状态 S 获取 B 面 9 个贴纸颜色 → 得到 "预期颜色直方图"
2. 从视频帧 F 的 B 面 ROI 提取 "实际颜色直方图"
3. 得分 = 直方图相关性 (Histogram Correlation)
```

**优点**: 不需要精确定位每个贴纸，只要颜色分布大致匹配。
**缺点**: 粒度粗，可能把不同状态评为相近分数。

#### 方案 B: 分区颜色匹配（中等难度，第二版）

```
1. B 面分成 3×3 = 9 个子区域
2. 每个子区域的主色 vs 虚拟魔方的对应贴纸颜色
3. 匹配 → 得分 +1，不匹配 → 得分 -1
4. 被遮挡区域 → 得分 0 (不奖不惩)
```

**前提**: 需要手动标定 B 面四角点（透视变换）。

#### 方案 C: 差分比较（最鲁棒，推荐并行尝试）

不比较绝对颜色，而是比较 **颜色变化**:

```
1. 虚拟魔方执行动作前后，哪些贴纸会变色？ → 预期变化区域
2. 视频前后帧差分 → 实际变化区域
3. 得分 = 预期变化区域 ∩ 实际变化区域 的重叠度
```

**优点**: 不依赖绝对颜色（避免光照影响），不需要精确定位贴纸。
**缺点**: 手部遮挡会产生虚假的"变化区域"。

> [!IMPORTANT]
> **推荐策略**: 初版同时实现方案 A 和方案 C，取加权平均。
> 方案 A 对"确定哪一面转了"有效，方案 C 对"验证是否合理"有效。

### 4.4 波束搜索引擎 (`search_solver.py`)

#### 数据结构

```
Path = {
    moves: ["U'", "R", "U2", ...],   # 已确定的动作序列 (从后往前)
    state: CubeState,                 # 当前虚拟魔方状态
    score: float,                     # 路径累积得分
}

Beam = [Path_1, Path_2, ..., Path_K]  # 当前保留的 K 条候选路径
```

#### 搜索过程

```
1. 初始化:
   - beam = [Path(moves=[], state=Solved, score=0)]
   - t = 最后一段 (index = 44, 对应 segment 45)

2. 逆序遍历每一段:
   FOR t = 44 → 0:
     new_beam = []
     FOR each path in beam:
       FOR each candidate_face in Top-K(visual_model(segment_t)):
         FOR each direction in [CW, CCW, DOUBLE]:
           move = candidate_face + direction   # 例如 R, R', R2
           new_state = path.state.apply_inverse(move)
           
           # 评分
           visual_score = score_visual(new_state, frame[t-1])
           cfop_score = score_cfop(path.moves, move, t)
           
           new_score = path.score + visual_score + cfop_score
           new_beam.append(Path(
             moves = path.moves + [move],
             state = new_state,
             score = new_score
           ))
     
     # 剪枝: 保留 Top Beam_Width
     beam = sorted(new_beam, key=score, reverse=True)[:BEAM_WIDTH]

3. 输出: beam[0].moves (reversed) 即为复盘结果
```

#### 参数选择

| 参数 | 初始值 | 依据 |
|------|--------|------|
| **Beam Width** | 20 | 候选面 5 × 方向 3 = 15 个分支/步，20 可保留至少 1 条正确路径 |
| **Top-K (视觉)** | 3 | 从概率分布取前 3 个候选面 |
| **方向扩展** | 3 | 每面 CW/CCW/DOUBLE |
| **每步候选数** | ≤ 9 | 3 面 × 3 方向 |
| **总路径扩展** | ≤ 180 | 20 beam × 9 候选 |

**计算量估算**: 45 步 × 180 次扩展 × (状态拷贝 + 评分) ≈ 8100 次操作。单次 < 1ms → 总计 < 10s。完全可接受。

## 五、CFOP 阶段约束

### 5.1 阶段检测

逆向搜索时，我们从 LL → F2L → Cross 的顺序逆推。可以根据虚拟魔方状态自动检测当前处于哪个阶段：

| 阶段 | 检测条件（逆推视角） | 约束 |
|------|---------------------|------|
| **Solved → LL** | 顶层不再完整/十字仍在 | 只允许 U + (R/L/F) |
| **LL → F2L** | 十字完整，F2L 槽位逐个被拆的过程 | 不应出现 D |
| **F2L → Cross** | F2L 槽位不完整 | 较宽松 |
| **Cross 阶段** | 底面十字不完整 | 所有面都可能 |

### 5.2 阶段得分

```
cfop_score(move, stage):
  if stage == "LL" and move.face == "D":
    return -10   # LL 阶段做 D 非常不合理
  if stage == "LL" and move.face in ["U", "R", "L", "F"]:
    return +1    # 鼓励 LL 常见的面
  ...
```

> [!TIP]
> 初版可以不做阶段约束，仅用视觉得分。阶段约束是后续提升的手段。

## 六、特殊情况处理

### 6.1 同时动作 (如 `UD'`)

GT 中存在 `UD'` 这样同时执行的动作对，在 splits 中算 1 段但是 2 个转动。

**处理方式**: 在候选生成时，除了单动作，额外生成常见的双动作组合：
- `UD'`, `U'D`, `U2D2` 等对面同时转动

### 6.2 宽转 (如 `r`, `f`)

`r = R + M'`（等价于旋转了两层）。

**处理方式**: 在候选集中加入 `r, r', f, f'` 作为独立候选。宽转的视觉信号比单层转更明显（变化区域更大），应该更容易识别。

### 6.3 整体旋转 (`x`, `y`)

- `y` 被单独分段 → 需要在候选集中加入 `y, y'`
- `x` 不被单独分段 → 和前一个动作捆绑出现（如 `L2'x'`）

**处理方式**: 
- `y` 的视觉信号非常明显（整个魔方旋转，所有面都变），通过视觉模型识别
- `x` 与前一个动作组合处理，增加组合候选（如 `L2'x'`）

### 6.4 卡顿 (`...`)

GT 中出现 `...`（如 `U'...f`），表示卡顿。在分段中 `...` 可能产生空段（帧差很小的"假静止"）。

**处理方式**: 如果某段的视觉变化低于阈值，标记为"空段"，搜索时跳过。

**验证方式**:

- 如果正确动作被排在后 50%，说明评分函数无效，需要重新设计
**预计耗时**: 2 天

## 十、需要的技术决策

以下几点需要在开发过程中逐步验证和决策：

1. **魔方库**: 用现有 Python 库还是自己写？
   - 现有库可能不支持宽转 `r`, `f` 或旋转 `x`, `y`
   - 自己写 200-300 行即可覆盖，且更可控
   - **建议**: 自己写，避免外部依赖和格式适配

2. **评分函数的权重分配**: 视觉得分 vs CFOP 得分的权重比
   - 初版: 100% 视觉, 0% CFOP
   - 逐步引入 CFOP 约束

3. **方向推断时机**: 在搜索内部扩展方向（3 倍分支）还是搜索出面后再定方向？
   - 内部扩展: 更准确, 但搜索空间 ×3
   - 后处理: 速度快, 但可能在关键步骤犯错
   - **建议**: 初版内部扩展 (计算量可控)

4. **同时动作的注入策略**: 何时尝试双动作？
   - 只在 `UD`, `U'D` 等对面组合时
   - 或从 GT 统计哪些段有同时动作，设计启发式

## 十一、分类器优化实验记录 (2026-02-14)

### 基线

CNN (ResNet18 512d) + HS 直方图差分 (96d) = 608d, PCA50 K=11 cosine = **LOO 57.6%**

### 实验结果

| # | 方向 | 准确率 | vs 基线 | 结论 |
|---|------|--------|---------|------|
| 1 | SVM (class_weight='balanced') | 59.4% | +1.8% | 微弱, 不值得引入 |
| 2 | R→L 水平翻转增广 | 50.2% | **-7.4%** | R 崩盘 (64%→23%), 镜像假设不成立 |
| 3 | 3×3 网格色差 (162d) | 58.1% | +0.5% | 微弱, F↑ 但 L↓ |
| 4 | R3D-18 3D CNN 时空特征 | 44.5% | **-13.1%** | Kinetics 预训练完全不适用 |
| 4b | ResNet + R3D 融合 (1024d) | 54.6% | -3.0% | R3D 特征拖累基线 |

### 关键发现

1. **ResNet18 单帧 + 直方图差分是当前最优**, 换分类器/加特征都改进甚微
2. **R3D-18 的 Kinetics 预训练完全不适用于魔方**, 通用动作识别和魔方转动差异太大
3. **镜像增广假设不成立**: 手部遮挡位置和相机不对称, R 翻转 ≠ L
4. **229 个样本 + 极端不平衡 (L:29, R:81) 是根本瓶颈**, 特征工程无法弥补

### 计算复杂度分析

| 阶段 | 复杂度 | 229 样本耗时 | 1000 样本预估 |
|------|--------|-------------|--------------|
| 特征提取 | O(n), 线性 | ~3 min | ~13 min |
| KNN LOO 评估 | O(n²), 二次方 | ~5 sec | ~100 sec |
| PCA + 超参搜索 | O(n² × 配置数) | ~30 sec | ~10 min |

**结论: <1000 样本时训练时间不是问题**, 瓶颈在 I/O (视频读取) 而非计算。
如果样本到 5000+, 可改用 K-Fold 代替 LOO 来降低评估复杂度。

### 下一步建议

| 方向 | 预期收益 | 代价 | 优先级 |
|------|---------|------|--------|
| **增加训练视频** | 高 (更多样本直接提升) | 标注时间 | ★★★★★ |
| **恢复光流 HOF** | 中 (+1.4% LOVO) | 计算慢 | ★★★ |
| **CFOP 结构约束后处理** | 中-高 (利用 91.7% Top-3) | 实现复杂 | ★★★★ |

### MLP 分类器实验 (同日)

#### 实现
- 在冻结的 608d 特征上训练 PyTorch MLP (608→256→5)
- 5-Fold CV 评估 (替代 LOO, 速度快 45 倍)
- 添加特征缓存 `videos/features_cache.npz`，后续运行跳过 ~3 分钟的视频读取

#### 超参搜索 (4 阶段, 24 种配置, 3-seed 平均)

| 阶段 | 搜索项 | 最优值 |
|------|--------|--------|
| Phase 1 | class_weight 强度 | cw=0.0 (不加权反而最好) |
| Phase 2 | hidden 层 | h=(256,) |
| Phase 3 | alpha + lr | a=1e-3, lr=2e-3 |
| Phase 4 | epochs | ep=500 |

**最优 MLP: h=(256,) cw=0 a=1e-3 lr=2e-3 ep=500 = 62.9% (3-seed avg 5-Fold CV)**

#### LOVO 泛化评估

| 方法 | LOO/5-Fold | LOVO | 差距 |
|------|-----------|------|------|
| KNN | 57.6% | 58.1% | +0.5% |
| MLP | 62.9% | 60.3% | -2.6% |

**MLP 泛化良好**, LOVO 仅降 2.6%。

#### 关键发现
1. **不加 class_weight 比全 balanced 好** — 全 balanced 过度补偿导致 U (88 样本) 从 78%→68%
2. **MLP 比 KNN 提升 +5.3%** (57.6% → 62.9%)
3. **模型没有过拟合特定视频** — LOVO 和 5-Fold 差距小
